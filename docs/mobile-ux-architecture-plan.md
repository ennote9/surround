# Mobile UX Architecture Plan

Архитектурный план мобильного UX для **Life Progress OS** (Vite + React + TypeScript). Документ относится к **блоку 22**, этап **22.1** — только проектирование, **без реализации UI и без изменений кода**.

---

## 1. Цель документа

Зафиксировать **стратегию** мобильного UX для Life Progress OS до начала разработки shell и адаптации страниц.

**Принципы:**

- **Desktop layout** сохраняет текущий **Sidebar** как основную навигацию.
- **Mobile layout** получает **нижнюю навигацию (bottom navigation)**.
- Bottom navigation — **только иконки**, **без видимых текстовых подписей** (labels).
- Доступность: подписи передаются через **`aria-label`** и **`title`** (и при необходимости другие ARIA-атрибуты).
- В рамках блока **22** на этом этапе **не меняются**: cloud, auth, persistence, схема Supabase.
- Порядок работ: сначала **оболочка / layout** (shell), затем **поэтапная адаптация страниц**.

---

## 2. Текущее состояние

- Приложение уже **cloud-first**: данные из **Supabase PostgreSQL**, сохранение в облако.
- **Auth gate** (ProtectedApp + CloudStateGate) работает; сессия и загрузка cloud state проверены.
- **Desktop**: основной layout построен вокруг **Sidebar** с основными разделами и **SaveStatusIndicator**.
- **MobileNav** в кодовой базе уже существует, но его нужно **переосмыслить** под новый mobile shell (bottom icon nav + More), чтобы не дублировать и не конфликтовать с нижней панелью.
- **Production**: `https://surround-eta.vercel.app`; SPA rewrites через `vercel.json`; прямые маршруты работают.

---

## 3. Breakpoint strategy

### Широкий экран (desktop / tablet)

- Использовать **текущий Sidebar**.
- Текущий **AppLayout** остаётся базовым для `md+` (или `lg+` — см. открытые вопросы).

### Узкий экран (mobile)

- **Скрывать** desktop **Sidebar** (визуально и для фокуса/таба — не дублировать навигацию).
- Показывать **bottom icon navigation**.
- Основной контент: **нижний padding**, чтобы контент и FAB не уезжали под панель.
- Учитывать **safe-area** на iOS (`env(safe-area-inset-bottom)` и при необходимости верхние inset).

### Рекомендация по breakpoint

- **Mobile** (bottom nav + без sidebar): **ниже breakpoint `md`** в Tailwind (условно до ~768px по ширине viewport).
- **Desktop** (sidebar): **`md+`**.

**Открытый вопрос:** если визуально комфортнее перенос границы на **`lg`**, зафиксировать после просмотра реальных макетов и контента (таблицы, широкие карточки).

## 4. Mobile bottom navigation

### Размещение и поведение

- Панель **внизу экрана**, **`position: fixed`** (или sticky в рамках единого shell), поверх контента с учётом z-index.
- **Только иконки** — без видимых текстовых labels под иконками.
- **5 основных пунктов**; активный маршрут должен читаться **визуально** (цвет / фон / форма).
- Каждый пункт: **`aria-label`**, **`title`**, достаточная **область нажатия** (см. §12).

### Структура маршрутов

| # | Раздел | Маршрут | Назначение |
|---|--------|---------|------------|
| 1 | Главная | `/` | Дашборд |
| 2 | Цели | `/goals` | Цели |
| 3 | Проекты | `/projects` | Проекты |
| 4 | Рутина | `/routine` | Привычки |
| 5 | Ещё | — | Открывает **More** (меню / sheet) со вторичными пунктами |

### Иконки (lucide / стиль shadcn)

Рекомендуемый набор:

1. **Главная** — `Home`
2. **Цели** — `Target`
3. **Проекты** — `FolderKanban`
4. **Рутина** — `CalendarCheck`
5. **Ещё** — `Menu` (альтернатива: `Ellipsis` / `MoreHorizontal`, если лучше визуально совпадёт с продуктом)

## 5. Почему без labels

- **Экономия вертикального места** под контент.
- **Меньше визуального шума** на узкой полосе.
- Ближе к привычным **мобильным паттернам** (Instagram / TikTok-класс: иконки в нижней полосе).
- При **пяти** пунктах текстовые подписи часто **перегружают** панель и режутся обрезкой.

**Компенсация:**

- `aria-label` + `title` для скринридеров и длинного тапа / hover где применимо.
- Чёткое **active** состояние и узнаваемые иконки.
- В пункте **«Ещё»** — меню с **текстовыми** пунктами для вторичных разделов.

---

## 6. Active state

**Направление (не жёсткие классы в коде):**

- **Неактивно:** приглушённый цвет, например muted slate (`text-slate-400` или аналог дизайн-системы).
- **Активно:** более контрастный цвет (`text-slate-950` или акцент `text-blue-600`) + опционально **мягкая плашка** (`rounded-full`, `bg-slate-100` / `bg-blue-50`).
- **Клавиатура / фокус:** сохранить **`focus-visible`** (outline / ring), не убирать только ради «чистоты».
- Активный маршрут для ссылок: **`aria-current="page"`** где уместно.

Визуал должен опираться не только на цвет (WCAG): комбинировать **иконку + фон/форма** для активного пункта.

## 7. More menu

Пункт **«Ещё»** открывает **вторичное меню**: список, **bottom sheet**, или компактный drawer — на этапе **22.2** достаточно **простого** варианта без тяжёлой анимации.

**Минимальный состав More:**

- **Аналитика** → `/analytics`
- **Настройки** → `/settings`
- **Аккаунт** → `/auth`

**Позже можно добавить:** Export/Import shortcuts, About, Help, статус синхронизации (если продуктово нужно).

## 8. SaveStatusIndicator на mobile

**Сейчас:** индикатор живёт в **Sidebar**, который на mobile будет **скрыт**.

**Задача:** не потерять обратную связь по сохранению в облако.

**Варианты размещения:**

1. Компактный **индикатор / точка** в верхней зоне (mobile header).
2. Пункт или блок статуса **внутри More**.
3. **Плавающий** мини-индикатор (осторожно с перекрытием контента).
4. Только **toast** при ошибке — недостаточно для «saved» без перегрузки.

**Рекомендация:**

- В **22.2** не внедрять сложный отдельный mobile save UX.
- В **22.3–22.x**: вынести статус в **More** и/или **компактно в mobile header**; ошибку сохранения показывать **явно**, не перегружая bottom nav.

---

## 9. Mobile header

**Цель:** ориентировать пользователя (заголовок раздела, опционально контекст цели) без дублирования всего desktop chrome.

**Возможности:**

- Заголовок текущей страницы / раздела.
- Опционально: **выбранная цель** (если продуктово критично на mobile).
- **Аккаунт** в header не обязателен, если **More** ведёт на `/auth`.

**Рекомендация:**

- На первом этапе mobile shell — **минимальный header** или аккуратное использование уже существующей верхней зоны, если она не конфликтует с bottom nav.
- Избежать **тройного дубля**: Sidebar + старый MobileNav + новая bottom nav — см. §10.

---

## 10. Existing MobileNav

**История:** MobileNav был полезен до появления целенаправленной **bottom icon navigation**.

**После внедрения bottom nav:**

- Либо **заменить** поведение MobileNav на части shell (header / sheet).
- Либо **отключить** на breakpoint mobile, если конфликтует с нижней панелью.
- Либо **удалить** после стабилизации нового shell (не делать «резко» в первом PR).

**Рекомендация:** в **22.2** добавить **BottomNavigation** и **аккуратно** отключить или заменить старый **MobileNav** на mobile при конфликте макета/доступности.

## 11. Page adaptation plan

### Dashboard (`/`)

- Карточки **в одну колонку**; снизу **отступ** под bottom nav.
- Снизить визуальную плотность; важные блоки **выше** fold по возможности.

### Goals (`/goals`)

- Карточки целей **touch-friendly**; кнопки не мельче разумного минимума.

### Projects (`/projects`) — приоритетный mobile-раздел

- Список и детали проекта: **stacked layout**; возможен **master/detail**: список → проект → назад.
- Группы и задачи: крупные зоны нажатия, понятная иерархия.

### Routine (`/routine`)

- Отметки привычек по дням — **крупные** toggle-зоны для пальца.

### Analytics (`/analytics`)

- Графики и таблицы: горизонтальный скролл только осознанно; возможно **упрощение** части графиков на узкой ширине.

### Settings (`/settings`)

- Секции **стеком**; import/export и опасные действия — читаемые на телефоне, с подтверждениями.

### Auth (`/auth`)

- Уже близка к одноколоночному виду; проверить **размеры полей**, отступы, поведение с **клавиатурой** (viewport / scroll).

---

## 12. Touch target rules

- Интерактивные элементы на mobile: ориентир **≥ 44×44 px** (Apple HIG) / не меньше комфортного минимума Material.
- Иконки bottom nav: **hit-area** больше видимой SVG (padding / invisible tap target).
- Деструктивные действия — **подтверждение** (alert / sheet).

---

## 13. Safe area / viewport

- Bottom nav: **`padding-bottom: env(safe-area-inset-bottom)`** (и при необходимости боковые inset).
- Основной scrollable контент: **`padding-bottom`** с запасом под высоту панели + safe area.
- Избегать **горизонтального overflow** на типовых страницах.
- **`100vh`**: использовать осознанно (mobile browser chrome); при проблемах — динамическая высота / `dvh` где поддерживается и оправдано.

---

## 14. Accessibility

- Nav items без видимых labels: обязательны **`aria-label`** (и **`title`** как дополнение).
- Активный маршрут: **`aria-current="page"`** на соответствующем элементе.
- Кнопка **More**: при раскрытии меню — **`aria-expanded`**, связь с меню через **`aria-controls`** при реализации.
- Сохранить **`focus-visible`** для клавиатуры.
- Не полагаться только на цвет для «active» (см. §6).

---

## 15. Что не меняем в блоке 22 (на уровне требований к этапам)

- **Схема Supabase** и **RLS** — не меняются ради mobile UX.
- **Auth flow** (логика входа) — без перестройки под mobile-only.
- **Cloud persistence**, **import JSON в облако** — без изменений в рамках mobile shell.
- **Vercel** / `vercel.json` workflow — без изменений.
- **seed JSON** — не меняется.
- **Desktop Sidebar** на `md+` — **не ломаем**, только дополняем mobile shell.
- **Production checklist / деплой-процесс** — не меняем из mobile-документа.
