# User Profile / Account Center Architecture Plan

## 1. Цель документа

Этот документ фиксирует **архитектуру пользовательского профиля** (Account Center) для **Life Progress OS** до начала реализации кода в блоке 23.

**Зафиксировано:**

- Страница **`/auth`** — это **шлюз аутентификации** (вход, регистрация, краткий статус при уже выполненном входе), а **не** полноценный личный кабинет.
- Нужен отдельный маршрут **`/profile`** — **Account Center** внутри залогиненного приложения.
- Профиль — часть **authenticated app** (тот же `RouterProvider` / `AppLayout`, что и остальные страницы после `ProtectedApp`).
- Профиль должен корректно работать на **desktop** и **mobile** (включая bottom nav после этапов блока 22).
- Профиль **не ломает** текущий поток **Supabase Auth** (signIn / signUp / signOut / session).
- Профиль **не меняет** модель **cloud AppState persistence** (цели, проекты, задачи и т.д. остаются как сейчас; профиль — отдельный слой метаданных пользователя).

Исходная схема БД и RLS для `profiles` описана в [docs/supabase-setup.md](supabase-setup.md) и [docs/database-schema-plan.md](database-schema-plan.md). **Схему в этом документе не меняем** — только планируем использование.

---

## 2. Текущее состояние

- **Supabase Auth** работает: регистрация, подтверждение email (если включено в проекте), вход, выход.
- **ProtectedApp** ограничивает основное приложение неаутентифицированным пользователям.
- **AuthPage** реализует signIn / signUp / signOut, использует **`window.location.assign`** для перехода на главную после входа (без `useNavigate`, т.к. страница может рендериться **вне** `RouterProvider`).
- При **залогиненном** пользователе AuthPage показывает **короткий блок**: «Вы уже вошли», email, «Перейти в приложение», «Выйти».
- В **Sidebar** и в **mobile More** пункт «**Аккаунт**» ведёт на **`/auth`** — фактически это смешение «аккаунт» и «auth gateway».
- Таблица **`public.profiles`** в Supabase уже предусмотрена (см. SQL в `supabase-setup.md`: `id`, `email`, `display_name`, `created_at`, `updated_at`), RLS-политики под «свою строку» описаны; **полноценный UI профиля** и репозиторий под чтение/запись display name в рамках продукта **ещё не реализованы** как единый Account Center.

---

## 3. Главный архитектурный выбор

### `/auth` — auth gateway

- Вход и регистрация.
- Сообщения об ошибках и состоянии «не настроен Supabase».
- При **уже выполненном входе**: краткий статус (email), кнопки **«Перейти в приложение»** (`/`), в будущем — **«Перейти в профиль»** (`/profile`), **«Выйти»**.
- Не дублировать длинные секции профиля на `/auth`.

### `/profile` — Account Center

- Доступен **только** аутентифицированному пользователю (маршрут под тем же деревом, что и `/settings`, с `ProtectedApp` + сессией).
- Отображение данных **аккаунта** (из Auth + при необходимости из `profiles`).
- Настройки **профиля** (в первую очередь **отображаемое имя** из `profiles.display_name`).
- Блок **синхронизации / статуса сохранения** (связь с облачным сохранением AppState — см. §15).
- **Безопасность**: сброс пароля по email, выход; расширения — позже.
- Ссылка на **`/settings`** для импорта/экспорта JSON и прочих настроек приложения — **без** переноса `DataManagementCard` на профиль.

## 4. Routing strategy

| Маршрут | Назначение |
|---------|------------|
| `/auth` | Auth gateway (вход / регистрация / краткий статус). |
| `/profile` | Account Center (будущая реализация). |
| `/settings` | Настройки приложения и данные (виджеты, группы, import/export). |
| `/` | Основное приложение (дашборд и навигация). |

**Поведение:**

- **Неаутентифицированный** пользователь на защищённых маршрутах видит **AuthPage** через `ProtectedApp` (как сейчас).
- **Аутентифицированный** пользователь может открыть **`/profile`**.
- После реализации блока 23: пункт **«Аккаунт»** в Sidebar и в mobile **More** должен вести на **`/profile`** (а не на `/auth`).
- **`/auth`** для залогиненного пользователя **остаётся доступным** (если остаётся в роутере): показывает **краткий** auth-status и ссылки на приложение и профиль — без дублирования всего Account Center.

## 5. Profile page structure (`/profile`)

Планируемая структура страницы (секции карточками, mobile — одна колонка).

### 5.1. Аккаунт

- **Email** (из `user.email` / сессии).
- **Отображаемое имя** (кратко; полное редактирование — в секции «Профиль»).
- **Провайдер** (если доступно из `app_metadata` / `user.identities` без утечки секретов).
- Статус **подтверждения email** (`email_confirmed_at` / флаги, безопасно для UI).
- **Создан аккаунт** (`user.created_at`).
- **Последний вход** (`user.last_sign_in_at`), если доступно из объекта пользователя Supabase.

### 5.2. Профиль

- Редактируемое поле **«Отображаемое имя»** (`profiles.display_name`).
- **Avatar** — только **future** (пока не планировать Storage).
- Кнопка **«Сохранить»** (см. §12).

### 5.3. Синхронизация

- Компактный статус сохранения облака (связь с существующим состоянием сохранения в приложении).
- Последнее успешное сохранение / время (если есть в state).
- Текст ошибки сохранения и кнопка сброса ошибки (если есть `clearCloudSaveError`).
- Пояснение: данные приложения **сохраняются в привязке к аккаунту** (облако).

### 5.4. Данные

- Кнопка или ссылка **«Управление данными»** → **`/settings`**.
- Короткая подсказка про **export/import JSON**.
- Предупреждение: **импорт в облако** заменяет данные; операция **не обязана быть транзакционной** на уровне UI — пользователь уже предупреждается в Settings.

### 5.5. Безопасность

- **Отправить письмо для сброса пароля** (Supabase Auth, см. §13).
- **Выйти** из аккаунта.
- **Future:** смена email, удаление аккаунта, 2FA, список активных сессий.

### 5.6. Приложение

- Отображение **production URL** (константа или env) для ориентира пользователя.
- **Версия / build** — future, если появится в продукте.
- **Plan / tier** — future.

## 6. Supabase Auth data

Из **Supabase Auth** (`User` / `Session`), в UI можно безопасно использовать типичные поля:

- `user.id` (UUID, совпадает с `profiles.id` и `user_id` в данных приложения).
- `user.email`.
- `user.created_at`.
- `user.last_sign_in_at` (если присутствует в типе/ответе SDK).
- `user.email_confirmed_at` (или логика «подтверждён / ожидает» на основе полей SDK).
- Провайдер: например из **`user.app_metadata`** / **`identities`** — только **отображение**, без секретов.

**Нельзя:**

- Показывать **access_token**, **refresh_token**, service keys.
- Хранить секреты в **localStorage** или в React state для отображения.

Источник данных — **`supabase.auth.getUser()`** / **`getSession()`** и подписка на **`onAuthStateChange`**, а не прямой SQL к `auth.users` с anon-ключом.

## 7. profiles table strategy

Фактическая схема в репозитории (см. [docs/supabase-setup.md](supabase-setup.md)):

```text
profiles (
  id uuid PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  email text,
  display_name text,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
)
```

**Принципы:**

- **`auth.users`** — источник истины для **идентификации и email** в смысле входа.
- **`profiles`** — **app-level** поля профиля: **`display_name`**, денормализованный **`email`** (если колонка используется и синхронизируется при необходимости — см. §23).
- **`display_name`** хранится в **`profiles.display_name`**.
- **`avatar_url`** в текущей схеме **нет** — добавление только отдельной миграцией и этапом.
- Создание строки профиля: **lazy upsert** при первом открытии **`/profile`** или при первом сохранении display name; альтернатива — **триггер после signup** (обсуждается в §23). Текущая документация в `supabase-setup.md` уже описывает оба подхода.

Перед реализацией **23.3** сверить актуальную схему в Supabase Dashboard с файлом `supabase-setup.md`.

## 8. profiles RLS

- Для **`profiles`** ожидаются политики вида: пользователь может **читать / вставлять / обновлять** только строку, где **`id = auth.uid()`** (как в `supabase-setup.md`).
- **Удаление** профиля с фронта **не планируется** на ранних этапах.
- **Удаление аккаунта** целиком — отдельный продуктовый и security-этап, не v1.

## 9. Repository plan

Планируемый файл:

**`src/shared/api/repositories/profilesRepository.ts`**

Функции (имена ориентировочные):

- **`getProfile(userId)`** — получить профиль текущего пользователя (`RepositoryResult<Profile | null>`).
- **`upsertProfile(profile)`** — создать или полностью задать строку (если понадобится для инициализации).
- **`updateProfile(userId, patch)`** — частичное обновление (например только `display_name`).

**Требования:**

- Возвращать **`RepositoryResult<T>`** в том же духе, что остальные репозитории проекта.
- **Не бросать** необработанные исключения наружу; ошибки — в `error` поля результата.

## 11. Profile types

Отдельные **доменные** типы (не путать с `User` из Supabase):

```text
Profile {
  id: string            // uuid, === auth user id
  email?: string        // опционально, из profiles или только из Auth в UI
  displayName?: string | null
  createdAt: string     // ISO
  updatedAt: string
}

ProfileUpdate {
  displayName?: string | null
}
```

Тип **`User`** Supabase использовать только в слое auth, не смешивать с **`Profile`** в одной модели «всё в одном».

## 12. Display name UX

- Поле **«Отображаемое имя»** с placeholder вроде: «Например, имя или ник».
- **`trim()`** перед сохранением.
- **Пустое значение** допустимо: в UI можно показывать **fallback на email** или «Без имени».
- **Длина** — разумный лимит (например **2–60** символов после trim; точное значение — при реализации + проверка на сервере / constraint).
- Кнопка **«Сохранить»**; состояния **loading / success / error**.
- **Optimistic update** на первом этапе **не обязателен** — можно обновлять UI после успешного ответа репозитория.
- Сохранение — через **`profilesRepository.updateProfile`** (или upsert при отсутствии строки).

## 13. Password reset / change password strategy

### Вариант A (рекомендуется для v1)

- Кнопка **«Отправить письмо для сброса пароля»**.
- Вызов **`supabase.auth.resetPasswordForEmail(email, { redirectTo })`**.
- **`redirectTo`** должен быть настроен для **production** (`https://surround-eta.vercel.app/...`) и **localhost** в Supabase Dashboard (Redirect URLs).
- **Реализация 23.4 (v1):** в коде `redirectTo` = `window.location.origin` + `"/auth"` (без хардкода домена). В Supabase Redirect URLs должны быть, например, `http://localhost:5173/auth` и продакшен-URL вида `https://…/auth`.

### Вариант B (future)

- Отдельная страница **«Новый пароль»** по ссылке из письма (recovery session).
- **`supabase.auth.updateUser({ password })`** после проверки сессии восстановления.

**Рекомендация:** сначала **23.4** — только отправка recovery email; полноценный смена-пароля-on-site — позже.

## 14. Sign out strategy

- На **`/profile`** (и при необходимости на **`/auth`**) кнопка **«Выйти»** вызывает **`signOut`** из существующего **AuthProvider** / `useAuth`.
- После выхода **`ProtectedApp`** снова показывает **AuthPage**; маршруты приложения недоступны без входа.
- **Не** очищать облачные таблицы вручную при logout; локальный/memory state приложения уходит при размонтировании согласно текущей архитектуре.

---

## 15. Save status / sync strategy

- На **desktop** индикатор сохранения уже в **Sidebar** (`SaveStatusIndicator`).
- На **mobile** глобальный индикатор — **known limitation** блока 22; в профиле можно дать **подробную** карточку статуса.
- Источник данных: существующее **cloud save state** из контекста приложения (например через **`useAppState`** / тот же источник, что питает `SaveStatusIndicator`), без дублирования бизнес-логики сохранения.
- Показывать: **готово / сохраняем / сохранено в HH:MM / ошибка** + при необходимости кнопка **`clearCloudSaveError`**.

**Рекомендация для v1:** компактная карточка «Синхронизация» на **`/profile`**; **не** переносить весь глобальный `SaveStatusIndicator` в header/mobile на этом же этапе.

## 16. Data management links

- Одна явная ссылка: **«Управление данными»** → **`/settings`**, с пояснением, что экспорт и импорт там.

## 19. AuthPage future changes

Когда пользователь **аутентифицирован**:

- Показать **email**.
- Кнопка **«Перейти в профиль»** → **`/profile`** через **`<a href="/profile">`** или **`window.location.assign('/profile')`** — **не** `Link` / **`useNavigate`**, если контекст рендера AuthPage **вне** `RouterProvider` сохранится.
- **«Перейти в приложение»** → **`/`** (как сейчас).
- **«Выйти»**.

Если в будущем AuthPage **всегда** окажется внутри Router — можно унифицировать на `Link`, но только после явной проверки дерева провайдеров.

## 20. Sidebar / Mobile More future changes

- **Залогинен:** пункт **«Аккаунт»** → **`/profile`** (Sidebar и mobile More).
- **Незалогинен** (если такой UI когда-либо появится в shell): **«Войти»** → **`/auth`**.
- В текущей модели **`ProtectedApp`** без входа основное приложение не показывается; тем не менее маршруты в коде должны оставаться **предсказуемыми**.

## 21. Что не делать в первой реализации

- Не реализовывать **удаление аккаунта**.
- Не реализовывать **смену email** в UI.
- Не добавлять **2FA**.
- Не показывать и не логировать **tokens**.
- Не вводить **billing / тарифы**.
- Не делать **загрузку аватара** и Supabase Storage для аватара.
- **Не менять** схему Supabase **без** отдельного миграционного этапа и сверки с продом.
- **Не переносить** `DataManagementCard` на профиль.
- **Не менять** модель **cloud AppState persistence** и существующий **import JSON** flow в Settings.
- Не ослаблять **RLS** ради удобства отладки.

## 24. Risks

- **Password reset** сломается при неверных **Redirect URLs** в Supabase.
- Прямой доступ к **`auth.users`** с **anon** невозможен — только **Auth API** и таблица **`profiles`**.
- Строка **`profiles`** может **отсутствовать** — UI и репозиторий должны делать **graceful** get + upsert.
- **Logout** должен корректно сбрасывать UI и не оставлять «залипший» cloud state без перезагрузки провайдеров (проверить при реализации).
- **Upsert** профиля должен уважать **RLS** и не пытаться писать чужой `id`.
- Любое отображение **PII** — минимально необходимое; не логировать email в сторонние сервисы из профиля.

## 25. Проверки для будущей реализации (после код-этапов)

- `npm run build` и `npm run lint`.
- Вход → открыть **`/profile`**.
- Отображаются **email** и read-only поля аккаунта.
- Сохранение **display_name** → refresh → значение сохранилось.
- **Logout** → снова AuthPage → вход снова работает.
- **Mobile**: профиль без горизонтального overflow; кнопки нажимаются.
- **Production** после деплоя: `/profile` и **recovery email** с production URL.

- Оставить **`/auth`** **шлюзом** аутентификации и короткого статуса.
- Внедрять поэтапно: **shell (read-only)** → **repository + display_name** → **password reset** → **навигация «Аккаунт»** → **polish**.
- **Удаление аккаунта** и тяжёлая security-функциональность — только после отдельного дизайна и аудита.
