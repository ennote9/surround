# Goal Deletion Behavior Plan

## 1. Цель документа

Этот документ фиксирует **безопасную модель удаления целей** в **Life Progress OS** до изменений кода и схемы БД (блок **24**, этап **24.1**).

**Зафиксировано:**

- Сейчас цель можно **архивировать** (и скрывать с дашборда), но **полноценного удаления цели** из доменной модели и облака как продукта **ещё нет** (или оно не подключено к UI/облаку end-to-end).
- Удаление цели **опасно**: цель связана с **projects**, **milestones**, переключателем **selected goal** и фильтрацией UI.
- Пользователь должен **явно выбрать** сценарий последствий (см. §3).
- В **cloud-режиме** удаление должно отражаться в **Supabase**, а не только в локальном **reducer** — иначе после перезагрузки данные «вернутся».
- Этот документ **не меняет** код, **не меняет** схему БД и **не меняет** RLS.

---

## 2. Текущее состояние

- **Goals** — верхний уровень доменной иерархии целей пользователя.
- **Projects** привязаны к цели через **`goal_id`** (в коде — `Project.goalId`).
- **Project groups** и **tasks** вложены в **projects** (группы → задачи).
- **Milestones** могут быть привязаны к **цели** (`goal_id`) **или** к **проекту** (`project_id`); в схеме зафиксирован **XOR**: ровно одно из полей не null (см. [docs/supabase-setup.md](supabase-setup.md), `milestones_scope_check`).
- **Habits** — отдельная сущность; в SQL есть опциональные **`goal_id` / `project_id`** с поведением FK при удалении цели/проекта (`on delete set null` для цели у привычки — см. актуальную схему в репозитории).
- **User settings**, **profiles** — не являются «детьми» цели в смысле этого плана.
- Есть поведение **архивации** цели.
- **Полноценного delete goal** (с диалогом, режимами и облачной согласованностью) **нет**.
- В **Settings** операции вроде **«Очистить текущие данные»** исторически ориентированы на **localStorage** и **не гарантируют** полную очистку облака; цели могут **оставаться или возвращаться** после reload — это **отдельный** hardening (см. §16).
- Документ [docs/database-schema-plan.md](database-schema-plan.md) уже предупреждает о конфликте **физического DELETE goals** с **`projects.goal_id ON DELETE RESTRICT`**.

---

## 3. Главный продуктовый выбор

При удалении цели пользователь выбирает один из двух режимов (оба доступны из **одного диалога удаления**):

| Режим | Кратко |
|--------|--------|
| **A** | **Удалить только цель** — проекты, группы и задачи **сохраняются**, но **отвязываются** от цели; вехи уровня цели для этой цели удаляются. |
| **B** | **Удалить цель вместе с проектами и задачами** — каскадное удаление доменных сущностей внутри цели. |

---

## 4. Режим A — удалить только цель

### Смысл

- Удаляется **запись цели** (`goals`).
- **Projects**, **project_groups**, **tasks** остаются в системе.
- **Вехи уровня проекта** (milestones с `project_id` в удаляемую цель не входят по определению — они привязаны к проекту) **остаются** вместе с проектами.
- **Вехи уровня цели** для этой цели (`goal_id = targetGoalId`, `project_id IS NULL`) — **удаляются** (не могут существовать без цели).
- **Habits** не удаляются как сущность; при наличии у привычки ссылки на цель поведение должно соответствовать FK в БД (см. §15 — **`on delete set null`** для `habits.goal_id` в текущем SQL репозитория).
- **User settings**, **profiles** — без изменений.

### Отвязка проектов

- Все **projects** с `goal_id = targetGoalId` (и `user_id` текущего пользователя) должны получить **`goal_id` = null** (или эквивалент «без цели» в доменной модели).

### UI copy (ориентир)

- Заголовок опции: **«Удалить только цель»**
- Пояснение: **«Проекты и задачи останутся, но будут отвязаны от этой цели. Вехи, привязанные только к цели, будут удалены.»**

### Технический смысл (логический порядок)

1. `UPDATE projects SET goal_id = NULL WHERE goal_id = :targetGoalId AND user_id = :userId`
2. `DELETE FROM milestones WHERE goal_id = :targetGoalId AND project_id IS NULL AND user_id = :userId` (или эквивалент под XOR-ограничение)
3. `DELETE FROM goals WHERE id = :targetGoalId AND user_id = :userId`

### Критический риск: несовместимость с текущей схемой

В [docs/supabase-setup.md](supabase-setup.md) для **`public.projects`** задано:

- **`goal_id uuid not null references public.goals(id) on delete restrict`**

То есть в **текущей** задокументированной схеме:

- **`goal_id` не nullable**;
- при попытке **`DELETE`** из **`goals`** пока есть проекты с этим **`goal_id`**, сработает **RESTRICT** (удаление цели **заблокировано**), если сначала не удалить/не переназначить проекты.

**Вывод для режима A:** перед реализацией необходима **проверка фактической схемы в Supabase** (prod/staging). Если **`goal_id` остаётся `NOT NULL`**, режим A **требует миграции** (сделать `goal_id` nullable и согласовать FK / `ON DELETE`), **либо** продуктово отказаться от режима A до миграции. **На этапе 24.1 схему не менять** — только зафиксировать риск и open question (см. §20).

**Обновление этапа 24.2:** выполнена read-only проверка подключённого проекта Supabase (MCP **`execute_sql`**). Подтверждено: **`projects.goal_id`** — **`NOT NULL`**; внешний ключ **`projects_goal_id_fkey`** — **`REFERENCES goals(id) ON DELETE RESTRICT`**. Подробности, таблица совместимости и **подготовленный** (не применённый на этапе 24.2) SQL — в [docs/goal-deletion-schema-check.md](goal-deletion-schema-check.md).

**Обновление этапа 24.3:** миграция **`make_projects_goal_nullable_for_goal_deletion`** применена через Supabase MCP (**`apply_migration`**); **`projects.goal_id`** стал **nullable**, FK — **`ON DELETE SET NULL`**. Факт и верификация — в [docs/goal-deletion-schema-check.md](goal-deletion-schema-check.md) §«Результат этапа 24.3».

## 5. Режим B — удалить цель вместе с проектами и задачами

### Удаляется

- Цель **`targetGoal`**.
- Все **projects** с `goal_id = targetGoalId` (и `user_id`).
- Все **project_groups** этих проектов.
- Все **tasks**, принадлежащие этим группам/проектам.
- **Milestones**:
  - с `goal_id = targetGoalId` (вехи уровня цели);
  - с `project_id IN (projectIds)` (вехи уровня удаляемых проектов).

### Не удаляется

- **Habits** и **habit_logs** как независимый домен (см. §6).
- Другие **goals** и **projects**, не принадлежащие удаляемой цели.
- **user_settings**, **profiles**.

### UI copy (ориентир)

- Заголовок опции: **«Удалить вместе с проектами и задачами»**
- Пояснение: **«Будут удалены цель, все проекты внутри неё, группы задач, задачи и связанные вехи. Это действие нельзя отменить.»**

### Технический порядок (от потомков к родителям)

Итоговый порядок зависит от **FK** в Postgres (каскады, `RESTRICT`). Логически:

1. Найти **`projectIds`** = `SELECT id FROM projects WHERE goal_id = :targetGoalId AND user_id = :userId`.
2. Удалить **tasks**, относящиеся к группам/проектам этих проектов (или полагаться на **`ON DELETE CASCADE`** с родителя, если он настроен — проверить фактическую схему).
3. Удалить **project_groups** для этих проектов (или каскад с `projects`).
4. Удалить **milestones**, где `goal_id = targetGoalId` **или** `project_id IN (projectIds)`.
5. Удалить **projects** из `projectIds`.
6. Удалить **goal** `targetGoalId`.

В текущем SQL репозитория у **`tasks`** и **`project_groups`** часто есть **`ON DELETE CASCADE`** от **projects** — это упрощает шаги, но **подтвердить** нужно по живой схеме и тестам.

## 6. Почему не удалять habits

- **Привычки** — отдельная доменная область (рутина), не вложенная в goal/project в продуктовом смысле блока целей.
- Удаление цели **не должно** уничтожать историю рутины без отдельного явного действия пользователя.
- Если в будущем появится жёсткая привязка привычек к цели/проекту в UI, это **отдельное** изменение модели и политик удаления.

---

## 7. Что делать с milestones

- **Веха уровня цели** не имеет смысла без цели → в **режиме A** удаляются вехи с `goal_id = targetGoalId` (и без `project_id` в смысле XOR).
- **Веха уровня проекта** живёт вместе с проектом → в **режиме B** удаляется вместе с проектом; в **режиме A** сохраняется.
- **Режим A:** удалить только goal-level milestones цели.
- **Режим B:** удалить goal-level milestones цели **и** project-level milestones всех удаляемых проектов.

---

## 8. Что делать с selected goal

Затрагивается:

- ключ **localStorage** (или иное хранилище) выбранной цели;
- **goal switcher** (Sidebar и др.);
- фильтрация на **Dashboard** / **Projects**.

**Поведение после удаления (продуктово):**

- Если удалённая цель была **выбрана** в UI → **сбросить** выбор на состояние вроде **«Все активные цели»** / **`null`**, чтобы не ссылаться на несуществующий id.
- **Режим A:** проекты остаются **без цели** → они должны быть **видны** при фильтре «все» / «без цели» (конкретное правило отображения — решение этапов **24.3–24.4**).
- **Режим B:** удалённые проекты **не отображаются** после успешного удаления и reload.

Реализация очистки **selected goal** в коде и миграция ключей — в этапах **24.3–24.4**, не в этом документе.

## 9. Cloud vs local behavior

- В **production / cloud** удаление должно выполняться в **Supabase** (через репозитории / сервис), иначе данные восстановятся при следующей загрузке.
- Локальный **reducer** обновляется **согласованно** с результатом облачной операции.
- **Optimistic update** допустим, но при ошибке нужен **явный** откат или перезагрузка состояния и сообщение пользователю.
- Для **деструктивного** удаления разумна **пессимистичная** схема: сначала успех в облаке, затем **dispatch** в локальный state (или единый слой «применить результат операции»).
- Текущая архитектура **action-based save** не должна **молча** игнорировать удаление цели: операция должна быть **явной** в слое синхронизации.

**Рекомендация:** ввести **явный** вызов доменного сервиса/репозитория удаления, затем применить результат к **AppState**, а не полагаться только на локальный `DELETE_GOAL` без облака.

## 10. Repository / API plan

### Варианты размещения логики

- **`goalsRepository`:** `deleteGoalOnly`, `deleteGoalWithProjects` — возможно, но операция затрагивает **несколько таблиц**.
- **Рекомендуется:** отдельный доменный модуль, например **`src/shared/api/goalDeletion.ts`** (или `goalDeletionService.ts`), который внутри вызывает существующие репозитории или клиент Supabase в правильном порядке.

### Функции (черновик контракта)

- **`deleteGoalOnly(userId, goalId): Promise<RepositoryResult<GoalDeletionResult>>`**
- **`deleteGoalWithProjects(userId, goalId): Promise<RepositoryResult<GoalDeletionResult>>`**

**`GoalDeletionResult` (метаданные успеха), например:**

- `deletedGoalId`
- `affectedProjectIds` (для режима A — отвязанные id; для B — удалённые)
- `deletedProjectCount` / `deletedTaskCount` (если дёшево посчитать)
- `mode: "goal-only" | "with-projects"`

**Требования:** не бросать **необработанные** исключения наружу; ошибки — в **`RepositoryResult`**.

## 11. Transaction / RPC consideration

**Риск:** последовательные вызовы **Supabase JS client** — **не одна транзакция** на сервере. При сбое посередине возможно **частичное** удаление / рассинхрон.

### V1 (клиентская последовательность)

- Проще внедрить.
- Риск **partial failure**.
- Нужны жёсткое **подтверждение** в UI, сообщения об ошибке и **QA** (§18).

### Future (RPC + транзакция)

- Функции Postgres, например **`delete_goal_only(user_id, goal_id)`**, **`delete_goal_with_projects(user_id, goal_id)`**.
- Одна **транзакция** на сервере → предпочтительно для production.

**Рекомендация:** честно заложить **V1** с предупреждением в UI и документации; для максимальной надёжности спланировать **миграцию на RPC** после стабилизации контракта.

## 14. Reducer / action plan (future, без реализации)

**Варианты:**

- Отдельные действия: **`DELETE_GOAL_ONLY`**, **`DELETE_GOAL_WITH_PROJECTS`**.
- Или одно: **`DELETE_GOAL`** с **`payload.mode`**: `"goal-only" | "with-projects"`.

**Рекомендация:** payload с **`mode`** + идентификатор цели.

**Reducer (логика):**

- **goal-only:** удалить цель из state; у релевантных проектов **`goalId` → `undefined` / `null`**; удалить из state **milestones**, привязанные только к этой цели.
- **with-projects:** удалить цель, все проекты с этим **`goalId`**, их группы и задачи, соответствующие milestones.

**Важно:** после ответа облака state должен **совпадать** с тем, что в БД.

## 15. Database constraints to verify (read-only, перед кодом)

Перед реализацией проверить в **Supabase Dashboard** или read-only SQL (на этапе **24.1 не выполнять** здесь):

1. **`projects.goal_id`**: nullable или **NOT NULL**? (в репозитории [supabase-setup.md](supabase-setup.md) — **NOT NULL** + **`on delete restrict`**.)
2. **FK** `projects.goal_id` → **`on delete`** поведение.
3. **`milestones`**: XOR **`milestones_scope_check`**, каскады на `goals` / `projects`.
4. **`tasks`**: связь с **`project_groups`** / **`projects`**, каскады.
5. **`project_groups`**: связь с **`projects`**, каскады.
6. **RLS:** разрешены ли **`update`** проектов (отвязка) и **`delete`** по `user_id` / `auth.uid()` для goals, projects, groups, tasks, milestones.

**Примеры read-only запросов** (для ручного запуска позже):

```sql
-- columns projects.goal_id
select column_name, is_nullable, data_type
from information_schema.columns
where table_schema = 'public' and table_name = 'projects' and column_name = 'goal_id';

-- FK from projects to goals
select conname, pg_get_constraintdef(oid)
from pg_constraint
where conrelid = 'public.projects'::regclass and contype = 'f';
```

## 16. Data Management Settings relation

- **Удаление цели** — отдельная **фича** с явным диалогом и режимами.
- **«Очистить текущие данные»** / сброс в **Settings** — отдельная **cloud clear / reset** операция.
- Оба относятся к **Data Management Hardening**, но **не смешивать** в одной кнопке без ясного UX.

**Future (например 24.5):** довести **Settings** clear/reset для **cloud**, чтобы после операции **цели и связанные сущности** не «возвращались» с сервера; полная очистка должна включать **goals** и зависимые данные по выбранному контракту продукта.

## 17. Safety requirements

- Любое **деструктивное** действие — с **подтверждением** и явным текстом последствий.
- Везде фильтрация по **`user_id`** / сессии; не трогать чужие строки.
- **Не** использовать **`service_role`** в клиенте.
- **Не** делать «тихих» удалений без feedback в UI.
- После успеха — обновить **UI** и **selected goal**.
- При ошибке — **понятное** сообщение; при partial failure (V1) — честное сообщение и путь «обновить данные» / поддержка.

## 18. QA checklist for future implementation

### Goal-only delete (после поддержки схемы / миграции)

- Создать цель с проектом и задачами.
- Удалить **только цель**.
- Цель исчезла из списка и из облака после reload.
- Проект остался; **`goal_id`** в БД **null** (если схема позволяет).
- Задачи на месте.
- Project-level milestones сохранились; goal-level для цели удалены.
- Refresh не возвращает цель.

### Goal with projects delete

- Создать цель с проектами, группами, задачами, вехами.
- Удалить **вместе с проектами**.
- Цель, проекты, группы, задачи и связанные milestones исчезли.
- Привычки не затронуты нежелательным каскадом (проверить FK на `habits`).
- Reload не восстанавливает удалённое.

### Selected goal

- Выбрать цель в switcher → удалить её → выбор сброшен; удалённая цель не отображается в switcher.

### Permissions

- Удаление только **своих** данных; нет ошибок RLS при корректном пользователе.

4. Показывать ли **счётчики** проектов/задач перед удалением?

6. Нужен ли отдельный режим фильтра **«Без цели»**?
7. Должна ли **Analytics** учитывать unassigned projects?
8. Что делать с **ключом selected goal** в **localStorage** при удалении (очистка vs миграция id)?
9. Нужно ли **логирование** деструктивных операций (audit)?
10. Показывать ли **напоминание об экспорте/backup** перед удалением?

---

## 21. Final recommendation

- Реализовать **оба** режима **A** и **B** с явным выбором в диалоге.
- **Режим A:** отвязать проекты (**`goal_id` → null**), удалить **goal-level milestones**, затем удалить цель — **только после** подтверждения, что схема БД это допускает; иначе сначала **миграция**.
- **Режим B:** каскадно удалить доменные данные цели в безопасном порядке с учётом FK / RPC.
- **Сначала** сверить **фактическую** схему и RLS.
- Для production-надёжности заложить переход на **RPC в транзакции**.
- **Settings** clear/reset для cloud — **отдельный** этап (**24.5**), не смешивать с удалением одной цели.
