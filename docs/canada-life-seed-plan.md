# Canada Life Progress Seed Plan

Документ описывает **проектирование** будущего импортируемого seed JSON для приложения Canada Progress OS / Life Progress OS. Это не сам JSON и не инструкция по изменению кода или `initialState`.

Соответствие модели данных: seed в будущем должен соответствовать структуре `AppState` в `src/store/appState.types.ts` — поля `projects` (`Project` → `TaskGroup` → `Task`), `habits` (`Habit`), `milestones` (`Milestone`), `settings` (`AppSettings`), а также полям проекта `showOnDashboard`, `statType` (`CharacterStatType`). Версионирование и миграции остаются за существующим механизмом приложения.

---

## Главная цель

Иммиграция в Канаду через развитие Android-профессии, получение степени **Bachelor’s in Computer Science** (в т.ч. через UoPeople), подготовку английского языка, накопление **оплачиваемого** профессионального опыта и последующую подачу через **Express Entry / STEM** или **PNP** (в т.ч. мониторинг BC PNP Tech, OINP Human Capital Priorities).

**Ограничитель:** закрытие **кредита** ориентировочно к **началу 2027 года**. До этого момента фокус не на дорогих иммиграционных шагах, а на **финансовой стабилизации**, языке, Android-обучении и портфолио.

**Стратегические этапы** (UoPeople, IELTS, ECA/WES, Express Entry) зафиксированы в проектах и milestones, но **не все стартуют сразу** — часть помечена как `later` или `strategic`.

**Универсальность приложения:** персональные данные пользователя не должны вшиваться в `initialState.ts`. Предпочтительный путь — **импорт JSON** через существующий механизм Import на следующих этапах.

---

## Проекты

Ниже — черновой дизайн проектов для seed. Для каждого: название, роль, `statType`, `showOnDashboard`, фаза активности (`active` / `later` / `strategic`), группы, черновые задачи.

> **Примечание по JSON:** у каждой сущности в runtime есть `id`, `createdAt`, `updatedAt` (и у задач привязка к `projectId` / `groupId`). При генерации seed их нужно будет заполнить уникальными значениями — в этом документе они не перечисляются.

Для первого seed **точные даты неизвестны** — на этапе генерации JSON нужно будет либо использовать условные-плейсхолдер даты (например далёкий future / дата создания seed), либо политику «без дедлайна» если приложение позволит; **даты будут добавлены позже** при появлении timeline.
