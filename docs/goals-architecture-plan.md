# Goals Architecture Plan

Этот документ описывает переход приложения к архитектуре Life Progress OS, где верхним уровнем становятся жизненные цели, а проекты являются частью конкретной цели.

## 1. Название и концепция

Life Progress OS — персональная система управления жизненными целями, проектами, задачами, привычками, прогрессом и аналитикой.

Canada Progress OS теперь трактуется как первый seed и первая крупная цель внутри Life Progress OS, а не как граница всего продукта.

Цель "Канада" — первая большая цель пользователя, но приложение должно поддерживать несколько крупных жизненных целей одновременно.

## 2. Почему нужна сущность Goal

Сейчас приложение работает вокруг проектов. "Общий прогресс" считается по всем проектам сразу. Это нормально, пока есть одна большая цель — Канада.

Но если появятся другие цели:

- здоровье;
- карьера;
- собственный продукт;
- финансы;
- образование;
- переезд / иммиграция;
- семья / отношения;

то общий прогресс станет неоднозначным.

Нужен верхний уровень:

Goal -> Project -> Group -> Task

Goal отвечает на вопрос:
"Ради какой большой жизненной цели существуют эти проекты?"

Project отвечает:
"Какое направление работы внутри цели нужно выполнить?"

## 3. Предлагаемая модель данных

Черновик сущности Goal:

```ts
export type GoalStatus = "active" | "later" | "archived";

export type Goal = {
  id: string;
  title: string;
  description?: string;
  targetDate?: string;
  status: GoalStatus;
  showOnDashboard?: boolean;
  createdAt: string;
  updatedAt: string;
};
```

Описание полей:

- `id` — стабильный идентификатор цели.
- `title` — название цели.
- `description` — смысл и рамки цели.
- `targetDate` — целевая дата большой цели, если она есть.
- `status`:
  - `active` — текущая цель;
  - `later` — цель на будущее;
  - `archived` — цель завершена или временно скрыта.
- `showOnDashboard` — показывать ли цель на Главной.
- `createdAt` / `updatedAt` — служебные timestamps.

Изменение в `Project`:

```ts
export type Project = {
  ...
  goalId?: string;
  ...
};
```

Пояснение:

- `goalId` связывает проект с целью;
- на этапе миграции все существующие проекты будут привязаны к `goal-canada`;
- для новых проектов после внедрения Goals `goalId` должен стать обязательным, но в модели может временно быть optional для совместимости со старыми импортами.

## 4. AppState v2

Будущая структура:

```ts
export type AppState = {
  version: 2;
  settings: AppSettings;
  goals: Goal[];
  projects: Project[];
  habits: Habit[];
  milestones: Milestone[];
};
```

Фиксация решения:

- переход на `version: 2` обязателен, так как меняется структура состояния;
- старые backup JSON v1 должны мигрировать в v2;
- import/export должен продолжить работать прозрачно для пользователя.

## 6. Selected Goal Scope и переключатель целей

В сайдбаре должен появиться переключатель текущей цели (GoalSwitcher), который задает рабочий контекст приложения.

Варианты выбора:

- Все активные цели
- Канада
- будущие цели пользователя

Выбранную цель хранить в localStorage, не в AppState.

Ключ:

```ts
export const SELECTED_GOAL_STORAGE_KEY =
  "canada-progress-os-selected-goal";
```

Значения:

- `"all"` — все активные цели;
- `goalId` — конкретная цель.

Mental model:

- `GoalSwitcher` = выбрать, в каком контексте работать.
- `GoalsPage` = управлять списком целей.

UI-note:

- в раскрытом сайдбаре показывать select/dropdown;
- в collapsed sidebar на первом этапе можно скрывать полноценный select или показывать компактный индикатор.

## 7. Как выбранная цель влияет на разделы

| Раздел | Все активные цели | Конкретная цель |
|---|---|---|
| Главная | агрегированный обзор активных целей | обзор выбранной цели |
| Цели | показывает все цели | показывает все цели, выбранная подсвечена |
| Проекты | проекты всех активных целей | проекты выбранной цели |
| Рутина | пока глобальная | пока глобальная |
| Аналитика | аналитика всех активных целей | аналитика выбранной цели |
| Настройки | глобальные | глобальные |

Пояснения:

- Рутину пока не фильтруем по цели.
- Настройки остаются глобальными.
- Архивные цели не участвуют в режиме "Все активные цели".

## 8. Character Stats в контексте цели

Принятое решение: статы персонажа зависят от selected goal.

Правила:

- `selected goal = all` -> статы считаются по проектам всех active goals;
- `selected goal = конкретный goalId` -> статы считаются только по проектам этой цели;
- `archived goals` не участвуют в глобальном режиме.

Пример:

Стат "Здоровье" может быть 0% внутри цели "Канада", но 45% внутри цели "Здоровье". Это корректно: статы показывают прогресс в текущем выбранном контексте.

## 9. Прогресс целей

Определения:

- Прогресс проекта = выполненные задачи проекта / все задачи проекта.
- Прогресс цели = выполненные задачи всех проектов цели / все задачи всех проектов цели.

Общий прогресс:

- если `selected goal = all` -> прогресс по всем проектам active goals;
- если `selected goal = конкретная цель` -> прогресс выбранной цели.

Тексты UI:

- `all` -> "Прогресс активных целей"
- конкретная цель -> "Прогресс цели"

## 10. Страница "Цели"

Нужна отдельная страница целей.

Навигация в сайдбаре:

- Главная
- Цели
- Проекты
- Рутина
- Аналитика
- Настройки

Назначение `GoalsPage`:

- создать цель;
- редактировать цель;
- архивировать цель;
- открыть цель как текущий `selected goal`;
- видеть прогресс цели;
- видеть количество проектов внутри цели.

Минимальная карточка цели:

Канада  
Активная · 9 проектов · 0%  
Целевая дата: не задана  
[Открыть] [Редактировать] [Архивировать]

Минимальная форма `GoalDialog`:

- Название
- Описание
- Целевая дата
- Статус
- Показывать на Главной

Пояснения:

- архивирование не удаляет проекты;
- архивные цели можно показывать на странице "Цели", но не включать в глобальный режим по умолчанию.

## 11. Dashboard после внедрения Goals

Если `selected goal = all`:

- Сводка: прогресс активных целей;
- блок "Цели" с карточками active goals;
- проекты: избранные/показываемые проекты из active goals;
- статы: по всем active goals.

Если `selected goal = конкретная цель`:

- Сводка: прогресс выбранной цели;
- проекты: только проекты выбранной цели;
- статы: только по выбранной цели;
- блок "Цели" можно скрывать или оставлять как компактный контекст.

Важно: на текущем этапе это только архитектурная фиксация, без реализации.

## 12. ProjectsPage после внедрения Goals

- список проектов фильтруется по `selected goal`;
- в режиме `all` показываются проекты всех active goals;
- при создании проекта:
  - если выбрана конкретная цель — новый проект получает ее `goalId`;
  - если выбран `all` — в `ProjectDialog` нужно выбрать цель вручную или использовать первую active goal как default;
- `ProjectDialog` получает поле "Цель".

## 13. Analytics после внедрения Goals

- аналитика фильтруется по `selected goal`;
- `ProjectTargetDatesCard` показывает проектные цели в выбранном scope;
- прогресс проектов и task status считаются по scoped projects.

Будущие возможные расширения:

- прогресс по целям;
- ближайшие цели;
- цели без `targetDate`;
- аналитика архивных целей.

## 14. Habits и milestones

Решение для первого этапа Goals:

- Habits остаются глобальными.
- Milestones остаются в текущей модели с привязкой к `projectId`.

Причины для habits:

- текущая модель не содержит `goalId`;
- фильтрация habits усложнит миграцию;
- часть привычек полезна сразу для нескольких целей.

Будущий вариант:

```ts
goalId?: string;
projectId?: string;
```

Для milestones в будущем:

```ts
goalId?: string;
projectId?: string;
```

Правило будущего уровня:

milestone может быть связан либо с целью, либо с проектом.

## 15. Миграция v1 -> v2

Алгоритм:

1. Если `state.version === 1`:
   - создать `goals: [goalCanada]`;
   - всем проектам без `goalId` поставить `goalId: "goal-canada"`;
   - сохранить settings, projects, habits, milestones;
   - вернуть `version: 2`.

2. Если импортируется v2:
   - валидировать `goals`;
   - валидировать `project.goalId`;
   - если `goalId` не существует, назначить fallback `goal-canada` или первую active goal.

3. Старые backup JSON v1 должны импортироваться.

4. Seed JSON обновить до v2 отдельным этапом.
