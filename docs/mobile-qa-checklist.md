# Mobile QA Checklist

Ручная проверка мобильной версии **Life Progress OS** после блока 22 (Mobile UX + PWA metadata). Используйте как регрессию перед релизом и после крупных изменений в layout или навигации.

---

## 1. Scope

### Что проверяется

- **Mobile shell** — `AppLayout`, отступы, отсутствие горизонтального overflow у страницы.
- **Bottom icon navigation** — панель, активные состояния, safe area.
- **More menu** — панель «Ещё», маршруты, закрытие, слои (z-index).
- **Dashboard** — сетка карточек, сводка, метрики, пустые состояния.
- **Goals** (`/goals`) — список целей, CRUD, длинные названия; отдельная глубокая адаптация Goals могла не входить в блок 22 — см. примечание в §7.
- **Projects** — список + детали, группы, задачи, диалоги.
- **Routine** — неделя, таблица привычек, чекбоксы по дням, диалоги.
- **Analytics** — карточки, графики, дедлайны, целевые даты.
- **Settings** — секции, данные, импорт/экспорт, сбросы.
- **Auth** — до и после входа, вне/внутри приложения.
- **PWA** — manifest, иконки, установка (без ожиданий offline).
- **Production** — деплой, прямые URL, env, Supabase redirect.

### Что не проверяется глубоко

- Внутренняя схема Supabase и SQL.
- Нагрузочное тестирование backend.
- **Offline** — service worker и кэш не включены (см. [docs/pwa-setup.md](pwa-setup.md)); без сети приложение не обязано работать.

## 3. Pre-checks

- [ ] Production открывается без ошибки DNS/SSL.
- [ ] Можно **войти** (Supabase Auth).
- [ ] После входа видны **цели / проекты / данные** (cloud или локально согласно конфигурации).
- [ ] В консоли **нет критичных runtime errors** при навигации по основным экранам.
- [ ] Сеть **online** (для Supabase).
- [ ] **Сессия** сохраняется после обновления страницы (в рамках ожидаемого поведения).
- [ ] **Прямой заход по URL** и **Refresh** на `/`, `/goals`, `/projects` не дают белый экран и не ломают роутер.

---

## 4. Mobile shell checklist

- [ ] На ширине **< `lg`** боковой **Sidebar** скрыт.
- [ ] **MobileNav** (заголовок «Life Progress OS») виден.
- [ ] **Bottom navigation** видна, закреплена снизу.
- [ ] В панели **только иконки**, без видимых текстовых подписей (допустим `sr-only`).
- [ ] Контент страницы **не перекрывается** нижней панелью (нижний padding у `main`).
- [ ] **Safe area снизу** — на устройстве с «чёлкой»/home indicator контент и иконки не прилипают к системной зоне.
- [ ] **Safe area сверху** — заголовок MobileNav учитывает `safe-area-inset-top`, где применено.
- [ ] **Нет горизонтального скролла** у всей страницы (лёгкий scroll внутри таблицы Routine или chart — ожидаемо по дизайну).
- [ ] **More** открывается по кнопке «Ещё».
- [ ] **More** закрывается: backdrop, Escape, пункт меню, повторное нажатие «Ещё» (по реализации).
- [ ] Кнопка **«Ещё»** при открытом More **кликабельна** (не перекрыта backdrop поверх панели навигации).
- [ ] **Active state** заметен у текущего маршрута и у «Ещё» на `/analytics`, `/settings`, `/auth`.
- [ ] У пунктов нижней навигации есть **`aria-label`** и **`title`**.

## 5. Bottom navigation routes

| Действие | Ожидание |
|----------|----------|
| Home | `/` |
| Goals | `/goals` |
| Projects | `/projects` |
| Routine | `/routine` |

| More → Настройки | `/settings` |

- [ ] **Refresh** на каждом из маршрутов выше — приложение загружается (404 нет).
- [ ] **Active** для главной только на точном `/` (если используется `end`).
- [ ] **More** визуально активна на префиксах `/analytics`, `/settings`, `/auth`.

## 6. Dashboard checklist

- [ ] Одна **колонка** на узком экране; двухколоночные блоки только с **`md`/`xl`** по макету.
- [ ] Заголовок «Главная» **не обрезается** странно (перенос/`break-words`).
- [ ] **OverallProgressCard** — текст, прогресс, ссылки без overflow.
- [ ] **MetricsGrid** — без overflow страницы; при 2 колонках на `sm` карточки читаемы.
- [ ] **ProjectSummaryCard** — длинные названия, метаданные.
- [ ] **TodayRoutinesCard** — читаемость.
- [ ] **Пустые состояния** (нет виджетов / нет проектов) — кнопки на всю ширину на mobile, текст переносится.
- [ ] Последний блок **не уходит под** bottom nav.
- [ ] **Touch** — кликабельные зоны ощутимо ≥ ~44px там, где кнопки/ссылки.

---

## 7. Goals checklist

- [ ] Список целей **читаем**, карточки без горизонтального overflow страницы.
- [ ] Создание / редактирование / архивирование цели.
- [ ] Кнопка **«Открыть»** (если есть) ведёт в проекты или нужный контекст.
- [ ] После выбора цели **контекст** (GoalSwitcher / главная) обновляется ожидаемо.
- [ ] **Длинное название** переносится или обрезается предсказуемо.

> **Примечание:** отдельный этап глубокой адаптации Goals под mobile мог не входить в блок 22. Если заметны проблемы — зафиксировать в Notes как **Partial** / **future polish**, не блокируя остальной QA.

---

## 8. Projects checklist

- [ ] На mobile: **список сверху**, **детали проекта ниже** (stacked).
- [ ] **ProjectList** — удобные строки/касания, выбранный проект заметен.
- [ ] **ProjectView** — шапка, мета, кнопки не ломают layout; на mobile кнопки в колонку при необходимости.
- [ ] **Развернуть все / Свернуть все** — две кнопки удобны по высоте.
- [ ] Группы **accordion** открываются/закрываются.
- [ ] Задачи: создание, редактирование, **toggle**, удаление.
- [ ] После **refresh** изменения на месте (store + cloud по дизайну).
- [ ] **ProjectDialog** — помещается по ширине и высоте, скролл внутри при необходимости.
- [ ] Нет **лишнего** horizontal scroll у всей страницы (горизонтальный scroll **только** внутри таблицы/графика — ок, если по ТЗ).

## 9. Routine checklist

- [ ] Таблица привычек **не раздувает** ширину страницы: горизонтальный scroll **только** в обёртке таблицы.
- [ ] **Чекбоксы** по дням удобно нажимать.
- [ ] Редактирование / удаление привычки.
- [ ] **Переключение недели** — три кнопки на mobile в колонку или удобный wrap.
- [ ] Создание привычки, **HabitDialog** на узкой ширине.
- [ ] После refresh **отметки** и список согласованы с сервером/локальным состоянием.
- [ ] **Удаление** привычки (диалог подтверждения читаем).

## 10. Analytics checklist

- [ ] **Summary cards** — одна колонка на очень узком, далее по сетке.
- [ ] **Bar charts** — при необходимости scroll **только** в области графика.
- [ ] **Pie chart** и **legend** не вылезают за карточку.
- [ ] **UpcomingDeadlines** — переносы, даты, статусы.
- [ ] **ProjectTargetDatesCard** — ссылки, проценты, даты.
- [ ] Пустые состояния без overflow.
- [ ] Нет горизонтального scroll **всей** страницы.

## 11. Settings checklist

- [ ] Страница читаема, секции **stacked**.
- [ ] **Select** режима групп — на всю ширину контейнера на mobile.
- [ ] **DashboardSettingsCard** — виджеты и статы, скролл внутри списка статов при необходимости.
- [ ] Кнопки **сброса** — достаточная высота и ширина на mobile.
- [ ] **DataManagementCard** — кнопки в колонку, **min-height** комфортный.
- [ ] **Экспорт JSON** — файл скачивается.
- [ ] **Импорт JSON** — диалог подтверждения и текст **читаемы**; длинные строки переносятся.
- [ ] **Очистка / сброс** — AlertDialog на узкой ширине, кнопки не обрезаны.
- [ ] Предупреждающие тексты **переносятся**.

## 12. Auth checklist

### До входа (часто вне `RouterProvider`)

- [ ] Карточка auth **не вылезает** по ширине (`px` у внешнего контейнера).
- [ ] Поля **email/password** на всю ширину.
- [ ] Переключение **Вход / Регистрация** удобно (две колонки или stack).
- [ ] Сообщения об ошибках и **Supabase не настроен** — переносы, `code`/env не ломают layout.
- [ ] Нет ошибки **Router context** (не использовать `Link`/`useNavigate` там, где нет Router — по текущей архитектуре).

### После входа (в т.ч. `/auth` внутри приложения)

- [ ] Email **break-all / truncate** безопасно.
- [ ] Кнопки «Перейти в приложение» / «Выйти» — **touch-friendly**, на mobile часто full width.
- [ ] **Logout** работает; после выхода **bottom nav не показывается**, если снова показывается только экран Auth вне shell.

## 13. Save status / sync checklist

- [ ] На **desktop** индикатор сохранения в Sidebar **виден** (если реализован).
- [ ] На **mobile** индикатор может **отсутствовать** — **известное ограничение** блока 22; зафиксировать в Notes.
- [ ] Создание/изменение сущностей **сохраняются** (облако/local по конфигурации).
- [ ] После **hard refresh** данные на месте.
- [ ] При ошибке сохранения пользователь видит feedback **где доступно** (toast / indicator на desktop).
- [ ] Отдельный UX для mobile save status — **future**.

## 14. PWA checklist

- [ ] Chrome → **Application → Manifest** — manifest загружен без ошибок.
- [ ] `name` / `short_name` соответствуют ожиданиям.
- [ ] Иконки **192** и **512** (и maskable) отдаются **200**, не 404.
- [ ] **`theme_color`** в DevTools совпадает с ожидаемым.
- [ ] **`display: standalone`** в manifest.
- [ ] **Android Chrome** — при выполнении критериев installability может появиться «Установить приложение» / Add to Home Screen.
- [ ] **iOS Safari** — «На экран Домой» подхватывает **apple-touch-icon**.
- [ ] **Offline** не ожидается: без сети страница может не загрузиться или Supabase упадёт — не считать дефектом PWA metadata.

## 15. Production deployment checklist

- [ ] На Vercel задеплоен **актуальный** commit.
- [ ] Статус deployment **Ready**.
- [ ] Прямые маршруты: `/projects`, `/settings`, `/routine`, `/analytics`, `/goals` — работают после refresh.
- [ ] В Vercel заданы **`VITE_SUPABASE_URL`**, **`VITE_SUPABASE_ANON_KEY`**.
- [ ] В Supabase **Redirect URLs** указаны production URL и при необходимости `http://localhost:5173`.

## 16. Accessibility checklist

- [ ] У пунктов bottom nav — **`aria-label`** и **`title`**.
- [ ] У активного маршрута — **`aria-current="page"`** (если реализовано через `NavLink`).
- [ ] **Focus visible** на интерактивах (клавиатура / TalkBack базово).
- [ ] У «Ещё» — **`aria-expanded`**, **`aria-controls`** при открытом меню.
- [ ] **Touch targets** около 44px для ключевых действий.
- [ ] **Контраст** основного текста и кнопок приемлемый.
- [ ] Критичный текст не **меньше** чем допустимо для чтения на mobile.

---

## 17. Known limitations

- **SaveStatusIndicator на mobile** — может быть недоступен; отдельный этап UX.
- **Offline / service worker** — не включены; приложение рассчитано на **сеть** и Supabase.
- **Импорт в облако** — замена данных не обязана быть транзакционной на уровне UI; пользователь предупреждён confirm-ом.
- **Master/detail** проектов на отдельных экранах без stacked list — возможное **future** улучшение.
- **Свой домен** — не часть этого чеклиста.
- **Native wrappers** (Capacitor и т.д.) — не делались.

## 19. Final criteria

Mobile QA можно считать **пройденным** (с оговорками в Notes), если:

1. Нет **горизонтального overflow** у основных экранов (допустимы изолированные внутренние scroll у таблицы Routine / графиков).
2. **Bottom nav** не перекрывает критичные действия и primary CTA внизу страницы.
3. Основные **CRUD** по проектам, задачам, привычкам, настройкам работают на mobile.
4. После **refresh** данные и маршруты ведут себя предсказуемо.
5. **Production** прямые маршруты открываются.
6. **Manifest** и иконки доступны (PWA metadata).
7. Нет **критичных** runtime errors в консоли при сценарии из чеклиста.

## 20. Как проверить installability (кратко)

1. **Chrome DevTools → Application → Manifest** — валидность, иконки.
2. **Lighthouse → PWA** — базовые пункты (полный 100 не обязателен без SW).
3. **Android** — Add to Home screen.
4. **iOS** — На экран «Домой», проверить иконку.
